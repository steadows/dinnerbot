# Project Blueprint: "DinnerBot" - Weekly Meal Planner & Grocery Assistant

## 1. Project Overview
**Goal:** Create a serverless application that runs weekly to generate dinner options using an LLM (Gemini), sends them via SMS to the family, waits for a selection, and replies with a consolidated grocery list.

**Infrastructure:** Google Cloud Platform (GCP)
**IDE/Assistant:** Cursor
**Language:** Python 3.10+

---

## 2. Architecture & Tech Stack

### High-Level Flow
1.  **Trigger:** Google Cloud Scheduler (Cron job: every Sunday at 2 PM).
2.  **Generator:** Google Cloud Function A (`generate_options`) calls Gemini API to create recipes.
3.  **Storage:** Google Firestore stores the generated options with a unique `session_id`.
4.  **Outbound:** Twilio API sends the options via SMS to user.
5.  **Inbound (Reply):** User replies via SMS. Twilio Webhook triggers Cloud Function B (`handle_reply`).
6.  **Processor:** `handle_reply` retrieves the session from Firestore, asks Gemini to convert the selection into a grocery list, and SMS replies back.

### Tech Stack
* **Compute:** Google Cloud Functions (2nd Gen)
* **Database:** Google Firestore (NoSQL, Serverless)
* **Scheduling:** Google Cloud Scheduler
* **LLM:** Google Gemini API (via `google-generativeai` lib)
* **Messaging:** Twilio (SMS)

### Data Models (Firestore)
**Collection:** `meal_sessions`
**Document ID:** `<auto-generated>`
```json
{
  "phone_number": "+1234567890",
  "created_at": "timestamp",
  "expires_at": "timestamp",
  "status": "pending_selection", // or "completed" or "expired"
  "options": {
    "1": {"name": "Spaghetti Carbonara", "ingredients": [...]},
    "2": {"name": "Tacos", "ingredients": [...]},
    "3": {"name": "Chicken Curry", "ingredients": [...]}
  },
  "selected_option": null
}
```

## 3. Security Considerations

### Twilio Webhook Validation
All incoming requests to the SMS webhook must be validated using Twilio's `RequestValidator` to prevent spoofed requests. The validator checks the `X-Twilio-Signature` header against the request URL and POST parameters.

```python
from twilio.request_validator import RequestValidator

validator = RequestValidator(auth_token)
is_valid = validator.validate(url, post_params, signature)
```

### Secrets Management
* **Local Development:** Use `.env` file (never commit to git).
* **Production (Cloud Functions):** Use **Google Secret Manager** to store sensitive credentials:
  - `GEMINI_API_KEY`
  - `TWILIO_ACCOUNT_SID`
  - `TWILIO_AUTH_TOKEN`

Access secrets at runtime via the Secret Manager client library or by mounting them as environment variables during deployment.

## 4. Implementation Phases (Step-by-Step)

### Phase 1: Environment Setup & Configuration
**Objective:** Set up the local environment and GCP project structure.
* **Tasks:**
    1.  Create `requirements.txt`:
        * `google-cloud-firestore`
        * `google-cloud-secret-manager`
        * `google-generativeai`
        * `twilio`
        * `flask`
        * `functions-framework`
    2.  Create `.env` template for local development (never commit real keys):
        * `GCP_PROJECT_ID`
        * `GEMINI_API_KEY`
        * `TWILIO_ACCOUNT_SID`
        * `TWILIO_AUTH_TOKEN`
        * `TWILIO_FROM_NUMBER`
        * `USER_PHONE_NUMBER`
    3.  Set up Google Secret Manager in GCP:
        * Create secrets for `GEMINI_API_KEY`, `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`.
        * Grant Cloud Functions service account access to read secrets.
    4.  Create `config.py` module to abstract secret loading (from env locally, from Secret Manager in production).
    5.  Set up `main.py` entry point structure for Cloud Functions.

### Phase 2: The Recipe Generator (LLM Logic)
**Objective:** Build the function that prompts Gemini for recipes.
* **Tasks:**
    1.  Create module `llm_service.py`.
    2.  Write function `generate_weekly_recipes()`:
        * Prompt Gemini: "Suggest 3 distinct dinner ideas for a family. Return ONLY valid JSON."
        * Enforce JSON schema validation.
    3.  Create module `db_service.py` to save these options to Firestore with `status: pending_selection`.

### Phase 3: Outbound Messaging (The Trigger)
**Objective:** Connect the Generator to Twilio and deployment.
* **Tasks:**
    1.  Create Cloud Function entry point `cron_trigger_recipes(request)`.
    2.  **Session Cleanup:** Before creating a new session, mark any existing `pending_selection` sessions for this user as `expired` to prevent stale session conflicts.
    3.  Call `generate_weekly_recipes()`.
    4.  Save session to Firestore with `phone_number`, `created_at`, and `expires_at` (e.g., 7 days from creation).
    5.  Format the 3 options into a readable SMS string (e.g., "1. Tacos\n2. Pizza").
    6.  Send SMS using `twilio.rest.Client`.
    7.  **Test:** Run locally and verify SMS is received.

### Phase 4: Inbound Handling (The Reply)
**Objective:** Handle the user's SMS selection.
* **Tasks:**
    1.  Create Cloud Function entry point `sms_webhook(request)`.
    2.  **Validate Twilio Signature:** Use `RequestValidator` to verify the request is from Twilio (see Section 3). Reject invalid requests with 403.
    3.  Parse incoming Twilio form data (`Body` text and `From` phone number).
    4.  Query Firestore for the session where `phone_number == From` AND `status == 'pending_selection'`. If no session found, reply with a friendly error message.
    5.  **Handle Invalid Input:** If user input doesn't match "1", "2", "3" (or recipe names), send an SMS asking them to reply with a valid option number.
    6.  Map valid user input (e.g., "1", "Option 2", "The Tacos") to the stored options.

### Phase 5: The Grocery List (Final Output)
**Objective:** Convert the selection to a shopping list.

**Why use LLM for grocery list?** While we store basic ingredients with each recipe, using Gemini to generate the final grocery list adds value by:
- Grouping items by store aisle (Produce, Dairy, Meat, Pantry, etc.)
- Suggesting appropriate quantities for a family
- Consolidating duplicate ingredients if multiple recipes are selected
- Formatting the list for easy reading via SMS

* **Tasks:**
    1.  Add function `generate_grocery_list(recipe_name, ingredients)` to `llm_service.py`.
        * Pass the stored ingredients as context to ensure consistency.
        * Prompt: "Given these ingredients: [ingredients], create a grocery list for [Recipe Name]. Group by store aisle and suggest quantities for 4 people."
    2.  Update `sms_webhook` to call this function after valid selection.
    3.  Send the final list back to the user via Twilio.
    4.  Update Firestore doc to `status: completed`.

## 5. Error Handling Strategy

### LLM Response Handling
* **JSON Parsing:** Wrap Gemini responses in try/except. If JSON is malformed, retry up to 2 times with a more explicit prompt ("Return ONLY valid JSON, no markdown").
* **Empty/Invalid Response:** If Gemini returns empty or unusable content after retries, use a fallback set of default recipes.

### Twilio Failures
* **Send Failure:** Log the error and store the message in Firestore with `status: send_failed` for manual review or retry.
* **Webhook Errors:** Always return a valid TwiML response, even on errors, to prevent Twilio from retrying indefinitely.

### User-Facing Error Messages
Keep error messages friendly and actionable:
* No active session: "Hi! I don't have any meal options waiting for you. Check back on Sunday!"
* Invalid selection: "I didn't catch that. Please reply with 1, 2, or 3 to select your meal."
* System error: "Oops, something went wrong on my end. Please try again in a few minutes."

## 7. Deployment Instructions (Reference)
* **Deploy Trigger Function:**
    `gcloud functions deploy trigger-weekly-meals --gen2 --runtime=python310 --trigger-http --entry-point=cron_trigger_recipes`
* **Deploy Webhook Function:**
    `gcloud functions deploy handle-sms-reply --gen2 --runtime=python310 --trigger-http --allow-unauthenticated --entry-point=sms_webhook`
* **Set Scheduler:**
    `gcloud scheduler jobs create http weekly-dinner-job --schedule="0 14 * * 0" --uri=[TRIGGER_FUNCTION_URL]`
